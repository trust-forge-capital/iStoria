# iStoria 竞品分析 v0.1

## 分析目标
- 明确 iStoria 在硬件监控赛道中的定位。
- 对比 GUI 菜单栏工具与 SSH 场景 CLI 的差异。
- 找出 iStoria 的首版差异化方向与应避免的正面硬碰硬。

## 结论先行
- iStoria 不应直接与 iStat Menus、Stats、TG Pro 争夺“桌面菜单栏 GUI 监控工具”定位。
- iStoria 应占据的空位是：**SSH 场景优先、单文件分发、跨平台、本机执行、终端友好 + JSON 友好** 的硬件监控 CLI。
- 对 Apple Silicon 的持续增强，是 iStoria 在 CLI 世界里的关键差异化点。
- 第一版不应该试图复制 iStat Menus 的天气、日历、世界时钟、超丰富 GUI 定制，而应把“统一命令体验 + 输出设计 + 自动化友好”做到极致。

## 竞品一：iStat Menus

### 定位
- macOS 菜单栏系统监控工具。
- 强调高可定制、覆盖面广、菜单栏常驻监控。

### 已观察到的核心能力
- CPU / GPU / 内存 / 磁盘 / 网络 / 电池 / 传感器 / 风扇 / 时间 / 世界时钟 / 天气。
- Rules / 通知能力较强。
- 支持风扇控制。
- 支持 Apple Silicon 更多频率与传感器信息。
- 强调低开销与大量菜单栏模式。

### 优势
- 功能覆盖最完整。
- 图形化体验成熟，适合长期常驻观察。
- 告警、天气、时间、组合模式等附加能力非常丰富。
- Apple Silicon 支持持续演进。

### 劣势 / 不适配点
- 主要是 GUI 产品，不是 SSH 工具。
- 主要面向 macOS，不是跨平台 CLI。
- 自动化与脚本友好性不是核心定位。
- 产品复杂度高，不适合“登录后快速命令查看”的轻量场景。

### 对 iStoria 的启发
- 可借鉴其“覆盖广 + Apple Silicon 增强 + 告警能力”的产品方向。
- 不要在第一版复制其 GUI 丰富度。
- CLI 版本更应强调统一入口和结构化输出。

## 竞品二：Stats

### 定位
- 开源 macOS 菜单栏系统监控工具。
- 定位接近轻量免费的 iStat Menus 替代方案。

### 已观察到的核心能力
- CPU / GPU / 内存 / 磁盘 / 网络 / 电池 / 传感器 / 蓝牙设备 / 多时区时钟。
- 提供风扇控制，但项目中明确标注该能力为 legacy / not maintained。
- 支持 Homebrew 安装。
- 为了检查更新与获取公网 IP，会访问外部 API。

### 优势
- 开源、免费、用户基础大。
- 功能覆盖较广。
- 对 macOS 用户吸引力强。
- 作为轻量菜单栏工具竞争力高。

### 劣势 / 不适配点
- 依然是 GUI 菜单栏产品，不是 SSH CLI。
- fan control 维护状态弱。
- 传感器映射尤其在 Apple Silicon 上存在天然复杂度，作者也明确提到新 SoC 需要时间适配。
- 不适合跨平台命令统一目标。

### 对 iStoria 的启发
- Apple Silicon 传感器命名与映射非常难，必须在 PRD 和技术方案中明确“增强但不承诺完全一致”。
- 开源工具也很难把所有硬件细节一次性做到完美，iStoria 第一版应先做稳定摘要。
- CLI 要避免沦为“只是把菜单栏数据换个壳输出”。

## 竞品三：TG Pro

### 定位
- 更偏硬件玩家/重度用户的温度监控与高级风扇控制工具。
- 重点在温度、风扇、过热保护、硬件侧可控能力。

### 已观察到的核心认知
- 官方定位明确强调 Temperature Monitoring & Advanced Fan Control。
- 相比综合型系统监控，它更突出“风扇控制”和“热管理”场景。

### 优势
- 在风扇控制、温度管理上品牌心智明确。
- 对高温、散热、硬件健康类用户价值清晰。

### 劣势 / 不适配点
- 不是通用 SSH 监控 CLI。
- 更偏单点深挖（温控/风扇）而非统一系统状态入口。
- 跨平台与脚本化不是核心卖点。

### 对 iStoria 的启发
- 如果 iStoria 将来要扩展高级传感器能力，可以借鉴其“热管理”价值叙事。
- 但第一版不应把自己定义为 fan-control 工具。
- iStoria 当前更适合做“观察”而不是“控制”。

## 竞品四：系统原生命令组合

### 典型命令
- macOS: `top`, `vm_stat`, `df`, `iostat`, `system_profiler`, `pmset`
- Linux: `top`, `free`, `df`, `vmstat`, `iostat`, `sensors`, `uptime`
- Windows: `wmic`, `systeminfo`, `Get-Counter`, `Get-CimInstance`, PowerShell 系列命令

### 优势
- 无需额外安装（部分命令系统自带）。
- 灵活、强大、脚本能力强。
- 对资深用户足够可用。

### 劣势
- 学习成本高。
- 跨平台不统一。
- 输出风格不统一。
- 用户为了回答一个问题，往往要组合多个命令。

### 对 iStoria 的启发
- iStoria 的真正替代对象之一，不是 GUI 工具，而是“分散的原生命令组合”。
- 产品价值必须围绕：
  - 统一入口
  - 降低认知负担
  - 输出一致
  - JSON 兼容自动化

## iStoria 的差异化定位

### 不做什么
- 不做桌面菜单栏监控 GUI。
- 不做 iStat Menus 的简化复刻版。
- 不做第一版多机集中控制台。
- 不做第一版复杂风扇控制器。

### 要做什么
- 做一个 SSH 场景优先的本机监控 CLI。
- 做一个统一入口，替代分散系统命令。
- 做一个既适合人读，也适合脚本调用的输出层。
- 做一个对 Apple Silicon 更友好的 CLI 监控工具。

## 产品机会总结

### 最大机会
- 目前强产品大多是 GUI 菜单栏方向，CLI 空位明显。
- 跨平台 + 单文件 + SSH-first 的组合很清晰。
- Apple Silicon 指标增强让 iStoria 有机会在 macOS 技术用户中建立辨识度。

### 最大风险
- 如果只做“系统命令二次封装”，价值会不足。
- 如果第一版就追求过度完整，容易在三平台一致性上失控。
- 如果过早承诺复杂告警、风扇控制、历史趋势，会拉长交付周期。

## 对产品路线的建议

### v0.1 应聚焦
- `stat`
- `cpu`
- `mem`
- `disk`
- `sensor`
- `--json`
- Apple Silicon 增强字段

### v0.2 可扩展
- `net`
- `power`
- 基础规则
- 本地告警记录

### 更后续再考虑
- 历史趋势
- 持续监控模式
- 高级通知通道
- 控制类能力（如风扇）

## 产品判断
- iStoria 的最佳切入点不是“比 GUI 工具更炫”，而是“比原生命令更统一，比 GUI 工具更适合 SSH 和自动化”。
- 如果做到这一点，iStoria 会有明确且健康的产品边界。
