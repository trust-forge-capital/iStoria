# iStoria 路线图与范围边界

## 一句话定位

iStoria 是一个适合 SSH 场景的跨平台硬件监控 CLI，帮助用户在登录目标机器后，用统一命令快速查看本机状态，并逐步获得规则与告警能力。

## 目标用户分层

### 核心用户（第一优先级）
- 开发者
- 运维工程师
- 高频使用 SSH 的技术用户
- 管理 Mac mini / Linux 主机 / 远程开发机的用户

### 次级用户（第二优先级）
- Apple Silicon 重度用户
- 需要脚本接入系统状态的自动化用户
- 喜欢命令行而不想依赖 GUI 菜单栏工具的用户

### 暂不优先用户
- 只想要桌面可视化面板的普通用户
- 需要企业级多机控制台的团队
- 需要复杂权限系统和云账户的组织用户

## 典型使用场景

### 场景 1：SSH 登录后快速看整机状态
用户登录远程机器后，执行 `istoria stat`，在一屏内看到 CPU、内存、磁盘、传感器、电源摘要。

### 场景 2：定位某台机器是否有资源瓶颈
用户通过 `istoria cpu`、`istoria mem`、`istoria disk` 快速判断瓶颈来源，而不需要组合多个系统命令。

### 场景 3：脚本自动采集状态
用户通过 `istoria stat --json` 获取统一结构化输出，接入 shell / Python / CI / 自动化脚本。

### 场景 4：Apple Silicon 专项观察
用户希望比通用系统命令获得更丰富的 Apple Silicon 指标，如频率、温度、性能核/能效核分组信息。

### 场景 5：实时持续观察当前状态
用户在 SSH 会话中希望像 `vnstat -l -i eth0` 一样持续看到当前状态变化，例如网络速率、CPU 波动、温度变化，而不是反复手动执行命令。

## 版本路线图

### v0.1：统一查看
目标：先解决“登录后快速看清当前机器状态”，并补充轻量实时刷新能力。

必须实现：
- `istoria stat`
- `istoria cpu`
- `istoria mem`
- `istoria disk`
- `istoria sensor`
- `--json`

应该实现：
- `istoria net`
- `istoria power`
- `--live` / `--watch` 实时刷新模式（优先支持 `stat` 与 `net`）

不做：
- 规则引擎
- 外部通知
- 历史趋势
- 持续后台模式
- 复杂 TUI / GUI

### v0.2：基础规则与告警
目标：从“查看”延伸到“发现异常”。

建议范围：
- `istoria rule add/ls/rm/test`
- `istoria alert ls/clear`
- 简单阈值规则
- 本地告警记录
- 非 0 exit code / 文本提示

### v0.3：自动化与持续监控增强
目标：增强脚本接入与长期使用价值。

建议范围：
- 更完整的 JSON schema
- 持续监控模式（如 watch/stream）
- 历史样本输出
- 更强的 Apple Silicon 指标覆盖

## 范围边界

### 明确要做
- 本机执行
- SSH-first
- 单文件分发
- 跨平台统一命令体验
- 人类可读输出 + JSON 输出

### 明确不做
- 多机集中控制台
- 云账户体系
- 第一版 GUI / 菜单栏应用
- 第一版高级风扇控制
- 第一版天气/日历/世界时钟等 GUI 型扩展能力

## 成功标准
- 首次用户 5 分钟内完成安装与首次执行。
- 用户可用 1~2 条 iStoria 命令替代过去的多条系统命令组合。
- `--json` 输出足够稳定，便于脚本复用。
- Apple Silicon 用户能明显感知指标覆盖优势。

## 产品提醒
- iStoria 不应成为“所有系统监控能力的集合体”。
- iStoria 应优先成为“SSH 环境下最顺手的硬件状态入口”。
